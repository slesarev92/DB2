# Решения заказчика — 2026-05-15

10 вопросов, отправленных 2026-05-12 после аудита MEMO v2.1, закрыты.
Решения ниже — основание для Фазы A/B/C работ.

---

## Q1 — Копакинг / Своё производство (MEMO 1.3, 3.2)

**Решение:** взаимоисключение per-SKU сохраняется, но **с возможностью
переключения по годам через Fine Tuning**.

**Пример из ответа:** Y1 — копакинг, Y2 — своё, Y3+ — копакинг.

**Гранулярность:** **годовая** (Y1..Y10), не помесячная.

**Что меняем в коде:**
- `ProjectSKU.production_mode` остаётся как **дефолт** (на случай если
  per-year не задан)
- Новая таблица или JSONB: `production_mode_by_year` с записями
  `(project_sku_id, year_index, mode)`, mode ∈ {"own", "copacking"}
- `s03_cogs.py` — взять mode для текущего года из per-year, fallback
  на скаляр SKU
- UI Fine Tuning: добавить строку "Способ производства" с
  переключателем own/copacking по 10 годам

**Инвариант:** в любом периоде ровно один режим. Копакинг и своё
производство **никогда не сосуществуют**.

## Q5 — Горизонт Y1-Y5: 5 или 6 лет? (MEMO 7.1.3, D-12)

**Решение:** **5 лет финально (Y1..Y5 = 60 месяцев)**, отсчёт от
начала проекта по конец 5-го года включительно. Аналогично:
- Y1..Y3 = 36 месяцев (с начала по конец 3-го года)
- Y1..Y10 = 120 месяцев (с начала по конец 10-го года)

**Что в коде:**
- `s11_kpi.py:50` `SCOPE_BOUNDS["y1y5"] = (5, 5)` — **остаётся**
- Комментарий "fix 6→5 одобрено 13.04.2026" — оставляем, актуален
- `docs/ROADMAP.md` "Технические долги" — отметка "Excel-тайпо
  сохранён намеренно" **устарела**, удалить или переподписать как
  "5 лет финально подтверждено 2026-05-15"
- D-12 в `docs/TZ_VS_EXCEL_DISCREPANCIES.md` — обновить статус

**Что НЕ делаем:** не возвращаемся к 6 лет (Y0..Y5), вариант MEMO
7.1.3 отклонён.

## Q6 — CA&M и Маркетинг: per-SKU или per-channel? (MEMO 3.2)

**Решение:** вариант **(а)** — перенести в **Каналы (per-channel)**.

**Обоснование:** маркетинг в HM/SM отличается от TT, унификация
per-SKU искажает экономику по каналам.

**Что меняем в коде:**
- БД: миграция — `ca_m_rate` и `marketing_rate` с `ProjectSKU` →
  `ProjectSKUChannel` (per-канал)
- Скрипт миграции данных: для каждой пары (SKU, канал) копируем
  текущее значение со SKU
- `s05_contribution.py` или `s07_below_gp.py` (где применяются ставки)
  — брать ставку с канала
- UI: убрать поля из карточки SKU, добавить в `channels-tab.tsx`
- Все существующие проекты пересчитать после миграции

**Цена:** ~6-8 часов работы + миграция.

## Q7 — НДС: ставка и применение (MEMO 3.3)

**Решение пересмотрено:** НДС в проекте используется **только для
пересчёта ex-factory от полочной цены** (формула D-02):

```
EX_FACTORY = SHELF_PRICE / (1 + VAT_RATE) × (1 − CHANNEL_MARGIN)
```

**Ставка управляемая** (поле `Project.vat_rate`). В РФ варианты:
- 22% — стандартная с 01.01.2026
- 20% — стандартная до 01.01.2026 (для старых проектов)
- 10% — льготная (отдельные категории)

**Что меняем в коде:**
- Дефолт `Project.vat_rate` → **0.22** (server_default миграция)
- Существующие проекты **не трогаем** (если 0.20 — пусть остаётся,
  это их история)
- UI: добавить поле "Ставка НДС" в карточку проекта с dropdown
  (22% / 20% / 10% / custom) и tooltip "используется для пересчёта
  ex-factory от полочной цены"

## Q8 — Per-ingredient НДС (MEMO 3.3)

**Решение пересмотрено:** **НДС из BOM исключается полностью**. Цены
сырья и упаковки вводятся **без НДС** напрямую.

**Что меняем в коде:**
- `BOMItem.vat_rate` — поле остаётся в БД (миграция drop = риск,
  смысла нет), **в движке не используется** (как сейчас)
- UI BOM: добавить пометку "цены без НДС" рядом с полями стоимости
- Документировать в карточке BOM / `docs/PATTERNS.md`: "BOM cost
  всегда без НДС; НДС применяется только при пересчёте
  shelf → ex-factory в s02_pricing"
- В будущем — можем дропнуть поле `BOMItem.vat_rate`, но не сейчас
