# B.9b — Помесячная гранулярность финплана (Y1-Y3) + UI 43 колонки

**Дата:** 2026-05-15
**Контекст:** B.9 (статьи CAPEX/OPEX) закрыта частично. Осталась
помесячная гранулярность: 36 месяцев Y1-Y3 + 7 лет Y4-Y10 = 43 точки.
**Источник:** MEMO v2.1 заказчика; план в `docs/CLIENT_FEEDBACK_v2_DECISIONS.md`.
**Сессия брейнсторма:** 2026-05-15, 4 ёмкие развилки закрыты.

---

## 1. Цель

Финансовый аналитик должен заполнять CAPEX и OPEX с **месячной точностью**
для первых 3 лет проекта — это покрывает фазу запуска, где cash flow
наиболее волатилен (закупка молдов, найм FTE, запускной маркетинг
по месяцам).

Y4-Y10 остаются годовыми — на этом горизонте месячная точность избыточна.

Итоговый горизонт: **43 периода** = 36 monthly + 7 yearly.

## 2. Что уже готово (нашли в брейнсторме)

**Engine — 0 изменений.** Расчётное ядро уже работает помесячно:

- `_load_project_financial_plan` (`calculation_service.py:103-135`)
  строит `tuple[float, ...]` длины 43, итерируя по всем периодам
  справочника — отсутствующие записи = 0.
- `s09_cash_flow.step` принимает `inp.capex` per-period, считает
  `ICF[t] = -capex[t]` для каждого из 43 t.
- `s10_discount.step` аннуализирует CAPEX обратно в годовые бакеты
  через `period_model_year` (строки 85-93). `annual_capex[Y1]` =
  Σ M1..M12 независимо от распределения.
- KPI (`s11_kpi`) работает на годовом уровне, scope (Y1Y3/Y1Y5/Y1Y10) — без изменений.
- Acceptance GORJI: не наполняет финплан → нули → KPI стабилен → drift 5% сохраняется.

**Таблица `capex_items` уже существует** (миграция `e5f6a7b8c9d0`),
UNIQUE на `(financial_plan_id, category, name)` — поддерживает
per-period записи.

**Таблица `opex_items` существует** аналогично.

**UNIQUE на `project_financial_plans (project_id, period_id)`**
гарантирует одну запись на (проект × период).

## 3. Решения брейнсторма

| Развилка | Решение | Обоснование |
|---|---|---|
| UI режим | Horizontal scroll, 43 колонки, sticky left column, группирующие заголовки Y1/Y2/Y3 над месяцами | Прямая аналогия Excel-листа финансиста. Меньше mental switching, нет двух UI-режимов. |
| Существующие данные | Lazy expand. БД не трогаем. UI показывает в M1/M13/M25 — то, что реально в БД. Bulk-fill «Распределить год» позволяет пользователю осознанно раскидать. | CAPEX и OPEX разные по природе (one-shot vs recurring). Универсального правила автомиграции нет — любой выбор испортит данные хотя бы для одной категории. |
| API контракт | Чистый `period_number` (1..43). `year` удаляется. GET всегда возвращает 43 элемента (отсутствующие = capex/opex/items = 0/[]). | DB2 — монолит, нет внешних консьюмеров API. Поддержка двух форматов = мёртвый код. |
| OPEX/CAPEX статьи | Каждая статья = одна строка с 43 ячейками. Авто-создание `OpexItem`/`CapexItem` при ненулевом значении, удаление при обнулении. Группы CAPEX/OPEX можно collapse/expand. | Это нативный для финансиста формат. Сохранение разбивки по статьям + по месяцам = смысл перехода. |

## 4. Архитектура

### 4.1. Слои

```
┌─────────────────────────────────────────────────┐
│ Frontend                                        │
│  financial-plan-editor.tsx (43 cols × N rows)  │
│  + bulk-fill toolbar (distribute year, fill)   │
└─────────────────────────────────────────────────┘
            ↓ HTTP, period_number 1..43
┌─────────────────────────────────────────────────┐
│ API: GET/PUT /api/projects/{id}/financial-plan │
│  Schema: FinancialPlanItem(period_number, ...)  │
└─────────────────────────────────────────────────┘
            ↓
┌─────────────────────────────────────────────────┐
│ Service: financial_plan_service.py              │
│  list_plan_by_period (43 элемента)              │
│  replace_plan (DELETE + INSERT по period_id)    │
└─────────────────────────────────────────────────┘
            ↓ ProjectFinancialPlan (PK), OpexItem/CapexItem (FK)
┌─────────────────────────────────────────────────┐
│ Engine (БЕЗ ИЗМЕНЕНИЙ)                          │

### 4.2. Backend изменения

**`backend/app/schemas/financial_plan.py`:**
- `FinancialPlanItem.year: int` → удалить.
- `FinancialPlanItem.period_number: int` — новое обязательное поле,
  Pydantic-валидация `1 <= period_number <= 43`.

**`backend/app/services/financial_plan_service.py`:**
- `_get_first_period_by_year` → удалить.
- Новый `_get_all_periods_by_number(session) -> dict[int, int]` —
  возвращает `{period_number: period_id}`.
- `list_plan_by_year` → переименовать в `list_plan_by_period`.
  Возвращает **43 элемента** (по одному на каждый `period_number`,
  отсутствующие в БД — `capex=0, opex=0, items=[]`).
- `replace_plan`:
  - Принимает массив до 43 элементов с `period_number`.
  - Маппит `period_number → period_id` через новый helper.
  - DELETE+INSERT логика без изменений (CASCADE для opex/capex_items).
  - Удалить логирование `item.year`, заменить на `item.period_number`.

**`backend/app/api/projects.py`** (или где endpoint живёт):
- Контракт `GET /api/projects/{id}/financial-plan` — отдаёт 43 элемента
  отсортированных по `period_number` (1, 2, ..., 43).
- Контракт `PUT` — принимает массив до 43 элементов. Валидация:
  `period_number` уникальны в массиве (Pydantic field_validator или
  model_validator), значения в диапазоне 1..43. Отсутствующие
  `period_number` в payload трактуются как «удалить запись» (DELETE+INSERT
  семантика replace_plan сохраняется).

### 4.3. Frontend изменения

**`frontend/components/projects/financial-plan-editor.tsx`:**

Структура:
```

│              ├──────┬──────┬──┤──────┬──────┬─┤    │     │     │

├──────────────┼──────┼──────┼──┼──────┼──────┼─┼────┼─────┼─────┤

Технические детали:
- **Сетка:** обычная HTML `<table>` с `overflow-x: auto` на родителе.
  AG Grid не используем — задача проще, чем pivot-таблицы (фиксированные
  колонки, прямой controlled state). 43 столбца × ~10 строк = 430 ячеек,
  React справляется без виртуализации.
- **Sticky:** первая колонка `position: sticky; left: 0`, первые
  две строки заголовков `position: sticky; top: 0`.
- **Группирующий заголовок Y1/Y2/Y3:** `<th colspan="12">` над M1..M12,
  M13..M24, M25..M36. Y4..Y10 — отдельные колонки одиночной шириной.
- **CAPEX/OPEX итог** — read-only ячейки = сумма ниже строк-статей за этот месяц.
- **«+ Статья»** — открывает popup для выбора `category` (из `CAPEX_CATEGORIES`/`OPEX_CATEGORIES`) и имени.
- **Collapse:** клик на иконку рядом с «CAPEX итог» / «OPEX итог»
  скрывает все её статьи (UI-only, состояние в `useState`).

**Bulk-fill toolbar (`frontend/components/projects/financial-plan-bulk-fill.tsx` — новый компонент):**

Два режима:
1. **«Распределить год»** — пользователь выбирает строку и год (Y1/Y2/Y3),
   указывает сумму X → UI записывает X/12 в каждую из 12 ячеек месяца.
   Для Y4-Y10 — режим неактивен (одна ячейка = один год).
2. **«Залить диапазон»** — выбор: строка, период «от» (`period_number` 1..43),
   период «до» (`period_number` 1..43), значение V → UI заполняет каждую
   ячейку значением V (recurring). Диапазон может смешивать monthly Y1-Y3
   и yearly Y4-Y10 (например, M1..Y10 = 40 ячеек) — семантика «значение V
   в каждый период» одинакова для monthly и yearly строк.

State: bulk-fill применяет изменения к React state редактора,
не отправляет в API напрямую — изменения сохраняются вместе с
кнопкой «Сохранить» editor'а (текущее поведение).

**Лоадинг существующих проектов:**
- При первом открытии редактора в новом UI: если все ненулевые записи
  лежат только в M1/M13/M25/Y4..Y10 (т.е. набор period_number ⊂ {1,13,25,37..43}) —
  показать toast/banner: «Этот финплан был сохранён годовыми точками.
  Используйте `Распределить год` для разнесения по месяцам.»
- Условие проверки делаем на фронте после `list_plan_by_period`.

## 5. Edge cases и риски

**E1. Существующий проект с 10 записями:**
- GET возвращает 43 элемента: ненулевые в M1/M13/M25/Y4..Y10, нули в остальных.
- Engine уже корректно работает (acceptance GORJI как proof).
- Banner подсказывает пользователю про bulk-fill.

**E2. Пустой проект (нет записей в `project_financial_plans`):**
- GET возвращает 43 нулевых элемента (включая `items=[]`).
- PUT с пустым массивом или массивом нулей → DELETE всех записей.

**E3. PUT с дублирующимися `period_number`:**
- Сервер отвечает 400. Pydantic-валидация: `period_number` уникален в массиве.

**E4. PUT с `period_number` вне 1..43:**
- Pydantic 422.

**E5. Статья с одинаковым `(category, name)` в разных периодах:**
- В таблице `capex_items` UNIQUE на `(financial_plan_id, category, name)`.
  `financial_plan_id` разный для разных периодов → нет конфликта.
- В UI: одна строка статьи представляет одно `(category, name)` через
  все 43 периода. Сохранение идёт `INSERT capex_items` для каждой ненулевой
  ячейки этой статьи в её `financial_plan_id`.

**E6. Округление при «Распределить год»:**
- X/12 даёт нецелое (X=100 → 8.333...). Используем `Decimal` или
  округление до 2 знаков. Последний месяц может корректироваться
  (X − 11×round(X/12, 2)) чтобы сумма точно = X.
- Решение: округление до 2 знаков для всех 12 ячеек; небольшое
  расхождение в total (≤ 0.06 ₽) допустимо. Пользователь видит и может
  поправить вручную.

**E7. Производительность frontend:**
- 43 колонок × ~10-20 строк = 430-860 controlled inputs. React 18 справляется,
  но при печати в одну ячейку идёт re-render всей таблицы. Если будет лагать —
  применяем `React.memo` на строку и `useCallback` на onChange. Не делаем
  заранее (YAGNI), но держим в уме.

**E8. Тесты:**
- Существующие backend-тесты `tests/services/test_financial_plan_service.py`
  (если есть) — обновить под `period_number`.
- Новые: тест что `list_plan_by_period` возвращает ровно 43 элемента.
- Новые: тест что `replace_plan` сохраняет в произвольный `period_number`.
- Frontend smoke: `tsc --noEmit` 0 ошибок, ручная проверка в браузере.

## 6. Acceptance criteria

1. Backend `pytest -q --ignore=tests/integration` — 472+ passed
   (новые тесты добавляют count, acceptance GORJI без изменения).
2. `npx tsc --noEmit` — 0 ошибок.
3. Ручная проверка:
   - Открыть существующий проект → редактор финплана отображает
     значения в M1/M13/M25 (старая семантика lazy expand). Banner виден.
   - Создать ненулевое значение в M5 для статьи «Аренда» → сохранить →
     перезагрузить страницу → значение видно в M5.
   - Bulk-fill «Распределить год» на Y1 для «Аренда» = 2400000 →
     12 ячеек по 200000 ₽ → сохранить → перезагрузить → значения видны.
   - Bulk-fill «Залить диапазон» M1..M36 для «Аренда» = 200000 →
     36 ячеек по 200000 → сохранить.
   - Удалить статью целиком → CASCADE удаляет все её per-period записи.
4. Расчёт KPI (доказательство invariance):
   - **Test A:** взять существующий проект с финпланом до B.9b. Прогнать
     `calculate_and_save_scenario`, запомнить NPV/IRR/ROI. Через новый
     API получить план (lazy expand → ненулевые в M1/M13/M25), сохранить
     обратно без изменений. Прогнать расчёт ещё раз — KPI **идентичны**
     до 6 знаков (нет тихих data losses при roundtrip).
   - **Test B:** создать проект, заполнить CAPEX = 15М ₽ в M3 (через новый
     UI/API). Прогнать расчёт. Создать тот же проект с CAPEX = 15М ₽ в Y1
     (используя «Распределить год» обратно: 1.25М × 12 ≈ 15М). Сравнить
     `annual_capex[Y1]` — оба = 15М. NPV/IRR одинаковы (аннуализация
     стирает разницу внутри года).
5. Миграции БД: alembic upgrade head → e5f6a7b8c9d0 (без новых миграций).
6. CHANGELOG.md обновлён в `[Unreleased]`.
